주간보고 전 전체 구조·실효성 점검. **코드/실제 호출로 확인, 추측·조작 없음, 변경 0(점검만).**
목적: "각 기능이 실제로 동작하는가 + 무엇이 실질 개선이고 무엇이 검증·규명인가"를 정직하게 구분.

> ⚠️ **수치 검증 원칙**: 보고서에 흔히 인용되는 일부 헤드라인 수치(법률 "78.6%→100%", BTI 도수 "1%→60%")는

## 1. KNN 전환 구조 — 실재 확인 (사실만)

읽은 코드: `scripts/train_knn.py`, `app/main.py /api/bti/feedback`, `app/core/survey_converter.py`(하이브리드 분기).

| 점검 항목 | 사실 |
|-----------|------|
| "데이터 쌓이면 학습→KNN, 없으면 룰" 구조 실재? | **예 (구현됨).** `survey_converter._load_knn_model()`가 `models/knn_bti_model.pkl` 있으면 로드, 없으면 룰. `determine_bti_code_hybrid()`가 모델 없으면 `('rule_based','medium')` 반환, 있으면 KNN vs 룰 비교(일치=hybrid_agree, proba≥0.7=knn, ≥0.4=rule_fallback, else rule). |
| 학습 트리거 | **자동 아님(수동).** `train_knn.py`를 사람이 실행 → pkl 생성 → **서버 재시작 시** 로드. 런타임 자동 재학습/핫스왑 없음. |
| 데이터 임계 | 코드 하한 **10개**(`load_feedback_data`: `<10`이면 학습 중단), docstring 권장 **50개+**. `is_correct=True`만 학습 데이터로 사용. |
| 지금 켤 수 있나? | **불가(의도된 상태).** `models/` 디렉터리 없음=pkl 없음. `data/bti_feedback.json`은 2건뿐(둘 다 test_user)이라 하한 10 미달. → **룰만 도는 게 정상.** |
| 헬스 확인 | `/health` 실호출 → `knn_model_loaded:false` (실측). 데이터 0~2건이므로 의도대로 룰 fallback. |
| 분위수 정규화 | **미구현.** BTI 분류 경로(`survey_converter`/`train_knn`)에 percentile/quantile/StandardScaler 없음(grep 확인). KNN 학습 시 스케일링도 미적용(실험용 `results.md`에서만 StandardScaler 비교). |

**결론(1)**: KNN/룰 하이브리드 **분기 코드는 실재**하나, **학습 데이터가 없어 현재는 룰 100%**. 분위수 정규화는 아직 없음. "구조만 준비된" 단계가 맞다.

## 2. 기능별 실효성 분류표

각 기능을 **[실질 개선 / 버그수정 / 검증·규명 / 기능 추가 / 구조만 준비]** 로 정직하게 분류. 근거는 코드·문서·실측.

### 2-1. BTI 분류 — 도수 임계 **버그수정** (분류 성능 향상 아님)
- **무엇**: alcohol(5번째 글자 H/L) 임계를 **9 → 5.5**로 변경(`API_GUIDE.md`: "도수 임계값은 5.5(과거 9에서 변경)"). alcohol은 실제 ABV가 아닌 0~10 파생 점수.
- **실측 효과(직접 계산)**: 임계 9에서는 H(고도수)가 거의 안 나오던 것이 5.5에서 정상 분포로 회복.
  - 실사용자 76명 alcohol: 임계 9.0 → H **0/76(0%)** / 임계 5.5 → H **19/76(25%)** (최댓값 8.95라 9 기준은 구조적으로 0%).
  - 합성 1504개: 임계 9.0 → **40.2%** / 5.5 → **64.6%**.
- **분류 정확도 자체는 향상 없음**: `results_bti_4vs5.md`·`results.md` — KNN이 룰 대비 합성에서 +3pp 수준, **hard 케이스는 오히려 룰이 우위**. 도수 글자는 빼면(4글자) 정확도가 오히려 오름(룰 0.763→0.900). 즉 이 수정은 **"고장난 글자를 정상 작동시킨 버그수정"**이지 분류 품질 개선이 아님.
- **분류**: 🔧 **버그수정**. (KNN 비교는 §2-7의 '검증·규명')
- ⚠️ 보고서 인용 "1%→60%"는 문서/git에서 미발견 → 위 실측(실 0%→25% / 합성 40%→65%)으로 대체.

### 2-2. 법률 필터 3등급(block/pass/review) — **실질 개선**
- **무엇**: 무검토 자동통과(0_auto_pass) 제거 + 펀딩 전용 판정(위험고지=정상신호) 정밀화. (`results_law_test.md`, 28케이스)
- **실측(펀딩 정밀화 전→후, 검증값)**: **안전율 96.4% → 100%(28/28)**, 엄격정확도 75.0%→82.1%, **위험 FN 0, 정상 자동차단 FP 1→0**, review 6→5.
- **분류**: ✅ **실질 개선**(안전성 지표 실측 상승).
- ⚠️ 보고서 인용 "78.6%→100%"는 현재 문서/git에서 미발견 → 검증값 **96.4%→100%**(또는 엄격정확도 75.0%→82.1%)로 보고 권장.

### 2-3. 법률 RAG — 설명 9개 → 조문 1,719개 **실질 개선**
- **무엇**: 9개 법령 "요약 설명" 검색에서 **실제 조문 단위 색인**으로 전환(`results_law.md`).
- **실측**: `data/law_articles.json` **1,719 레코드**, chroma `rag_db/law/law_articles` **count=1,719**(실호출 확인). PersistentClient로 영속, 9개 설명 컬렉션은 Ephemeral fallback.
- **분류**: ✅ **실질 개선**(검색 단위 9→1,719, 조문 인용 가능).
- 한계(정직): MiniLM 다국어 경량모델이라 일부 쿼리 의미검색 부정확(예 "숙취 해소") — 키워드 1차탐지로 보완. ko-sbert 전환은 향후과제.

### 2-4. 이미지 생성 — 입력별 차별화 **실질 개선**
- **무엇**: 고정 프롬프트가 아니라 **맛벡터→색·질감, 재료→소품, 지역→배경, 입력해시→구도/조명/스타일 회전**으로 술마다 다른 프롬프트 생성(`app/image_generator.py` 확인).
- **분류**: ✅ **실질 개선**(입력 구동 차별화 로직 실재).
- 한계: 백엔드가 아직 taste_vector를 전달하지 않으면 기본값 사용(향후과제 4-1).

### 2-5. 농사로(재료-지역) — 하드코딩 → 실데이터 **실질 개선**
- **무엇**: 재료별 지역 매핑·지역특산물을 수집 데이터로 사용.
- **실측**: `data/ingredient_region_map.json` **226개 매핑**(dict), `data/local_products.json` **415개 상품**. 수집 json 없을 때만 하드코딩 fallback(`results_audit.md` L 항목과 일치).
- **분류**: ✅ **실질 개선**(실수집 데이터 226/415건 사용).

### 2-6. OCR — 판별 구조화 + 검증 **검증·규명** (위조탐지 약점)
- **무엇**: 서류 레지스트리(`SUPPORTED_DOC_TYPES`) 기반 판별 + positive/negative 혼동행렬 검증(`results_ocr.md`).
- **실측**: 정상 판별 5/5, **무관/비서류 3종 정확히 거름**, **위조 의심 1건 False Accept**(식별번호 훼손 서류를 valid 처리), 발급일/개업일 혼동(#1) 재확인.
- **분류**: 🔎 **검증·규명**(구조화는 개선이나 이번 작업의 핵심은 검증). **위조 탐지 약점 명시** — 구조가 서류 같으면 통과시킴.

### 2-7. 인사이트 — preferences(선호 분포) **기능 추가**
- **무엇**: 기존 통계/예측/클러스터에 더해 **BTI 분포·8축 평균·음식/향/과일 선호 빈도** 집계 추가(`app/insight.py` `preferences`, `_aggregate_preferences`).
- **실측**: `/api/insight` 실호출 → 응답에 `preferences` 키 존재(`data_source: memory`=DB 없을 때 샘플 fallback).
- **분류**: ➕ **기능 추가**(신규 인사이트 항목).

### 2-8. (참고) KNN vs 룰 비교·오분류 진단·설문/임계/견고성 점검 — **검증·규명**
- KNN 비교(`results.md`,`results_bti_4vs5.md`): **분류 성능 향상 없음**을 규명(룰이 경계에서 우위).
- 오분류 진단(`results_misclass.md`): 오분류 91%가 1글자=경계노이즈, 룰 모순 0 → "정상 범위" 규명.
- 설문분석/임계점검/견고성(`results_survey_analysis.md`/`results_threshold_check.md`/`results_audit2.md`): 실데이터로 한계·견고성 규명.
- **분류**: 🔎 **검증·규명**(서비스 능력 변화 없음, 근거 확보).

## 3. 단맛 82% 쏠림 — 해석 (둘 다 가능, 76명으론 구분 불가)

- **실측**(`results_threshold_check.md`): 임계 5.0에서 sweetness가 **S 82%(62/76) : D 18%(14/76)** 로 가장 심하게 쏠림. 네 룰축 중앙값이 모두 5.0 초과.
- **두 가지 해석이 모두 가능**:
  1. **표본 편향**: 지인 중심 비무작위 소표본(N=76)이라 단맛 선호자가 과대표집됐을 수 있음.
  2. **모집단 특성**: 한국 막걸리/전통주 입문층의 단맛 선호가 실제로 일반적일 수 있음(달콤한 막걸리 인기).
- **76명으로는 둘을 구분할 수 없음** → 어느 쪽이든 **고정 임계 5.0이 이 데이터를 잘 못 가른다**는 사실은 동일. 
- **한계/필요**: 연령·지역·음용경험이 다양한 **더 큰 무작위 표본**이 있어야 편향 여부 판별 가능. 그 전엔 임계 조정 보류(향후과제), 단맛 축 쏠림이 최우선 점검 대상.

### 정직한 총평
- **실질 개선 4 + 기능 추가 1**: 법률(3등급·조문RAG)·이미지·농사로가 실제 능력 향상. 인사이트는 신규 항목 추가.

- **검증·규명 다수**: 이번 사이클 산출물의 상당수는 "성능 향상"이 아니라 **동작/한계의 정직한 규명**(KNN 이득 미미, 위조탐지 약점, 임계 5.0 부적합, 견고성 양호).
- **구조만 준비**: KNN 전환·REVIEW 큐·이미지 맛벡터 연동·분위수 정규화는 코드/응답은 있으나 **데이터·백엔드 연동 대기**.
- **수치 정직성**: "78.6%→100%", "1%→60%" 등 일부 인용 수치는 문서로 재현 불가 → 검증값으로 대체해 보고.

*(점검만 수행. 코드·데이터 변경 0. 서버는 점검 후 종료, 임시파일 정리 완료.)*
